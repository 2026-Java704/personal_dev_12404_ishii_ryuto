# お薬チェッくん README

## 1. アプリ概要

このアプリは、ユーザーごとに服薬情報を管理するためのWebアプリケーションです。

ユーザーは新規登録・ログインを行い、自分が服用する薬の名前、個数、備考、服薬済みかどうかを登録できます。登録した薬は一覧画面で確認でき、更新・削除もできます。

---

## 2. 主な機能

| No | 機能 | 内容 |
|---|---|---|
| 1 | ログイン | ユーザー名とパスワードでログインする |
| 2 | ログアウト | セッション情報を破棄してログイン画面へ戻る |
| 3 | 新規ユーザー登録 | ユーザー名とパスワードを登録する |
| 4 | 服薬一覧表示 | ログイン中ユーザーの薬情報を一覧表示する |
| 5 | 薬情報登録 | 薬の名前、個数、備考を登録する |
| 6 | 薬情報更新 | 登録済みの薬情報、服薬済み状態を更新する |
| 7 | 薬情報削除 | 登録済みの薬情報を削除する |

## 7. データベース設計

### users テーブル

ユーザー情報を管理するテーブルです。

```sql
CREATE TABLE users (
    id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name VARCHAR(40) NOT NULL,
    password VARCHAR(255)
);
```

| カラム名 | 型 | 制約 | 内容 |
|---|---|---|---|
| id | INTEGER | PRIMARY KEY / 自動採番 | ユーザーID |
| name | VARCHAR(40) | NOT NULL | ユーザー名 |
| password | VARCHAR(255) | なし | パスワード |

---

### medicine テーブル

薬情報を管理するテーブルです。

```sql
CREATE TABLE medicine (
    id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name VARCHAR(40) NOT NULL,
    note VARCHAR(500),
    count INTEGER,
    m_check boolean NOT NULL DEFAULT FALSE,
    users_id INTEGER REFERENCES users(id)
);
```

| カラム名 | 型 | 制約 | 内容 |
|---|---|---|---|
| id | INTEGER | PRIMARY KEY / 自動採番 | 薬ID |
| name | VARCHAR(40) | NOT NULL | 薬名 |
| note | VARCHAR(500) | なし | 備考 |
| count | INTEGER | なし | 個数 |
| m_check | boolean | NOT NULL DEFAULT FALSE | 服薬済みかどうか |
| users_id | INTEGER | FOREIGN KEY | ユーザーID |

## 12. 処理の流れ

### 12-1. ログイン処理

```text
ログイン画面を表示
  ↓
ユーザー名とパスワードを入力
  ↓
UsersRepositoryでユーザー検索
  ↓
一致するユーザーが存在するか確認
  ↓
存在する場合、Accountにidとnameを保存
  ↓
服薬一覧画面へ遷移
```

### 12-2. 新規ユーザー登録処理

```text
新規登録画面を表示
  ↓
ユーザー名とパスワードを入力
  ↓
入力チェック
  ↓
問題がなければusersテーブルへ保存
  ↓
ログイン画面へ戻る
```

### 12-3. 服薬一覧表示処理

```text
服薬一覧画面へアクセス
  ↓
Accountからログイン中ユーザーIDを取得
  ↓
MedicineRepositoryでユーザーIDに紐づく薬情報を検索
  ↓
medicine.htmlへmedicineListを渡す
  ↓
一覧表示
```

### 12-4. 薬新規登録処理

```text
薬新規登録画面を表示
  ↓
薬名、個数、備考を入力
  ↓
Accountからログイン中ユーザーIDを取得
  ↓
UsersRepositoryでユーザー情報を取得
  ↓
Medicineを作成
  ↓
medicineテーブルへ保存
  ↓
服薬一覧画面へ戻る
```

### 12-5. 薬更新処理

```text
服薬一覧画面で更新ボタンを押す
  ↓
対象IDの薬情報を取得
  ↓
更新画面へ現在の値を表示
  ↓
薬名、個数、服薬済み状態、備考を修正
  ↓
medicineテーブルを更新
  ↓
服薬一覧画面へ戻る
```

### 12-6. 薬削除処理

```text
服薬一覧画面で削除ボタンを押す
  ↓
対象IDの薬情報を削除
  ↓
服薬一覧画面へ戻る
```

## 13. application.properties

現在のDB接続設定は以下です。

```properties
spring.datasource.driver-class-name=org.postgresql.Driver
spring.datasource.url=jdbc:postgresql:personal_dev_12404_ishii_ryuto
spring.datasource.username=student
spring.datasource.password=himitu
spring.jpa.show-sql=true
spring.sql.init.mode=always
```

### 注意点

`spring.sql.init.mode=always` のため、アプリ起動時に `schema.sql` と `data.sql` が実行されます。

開発中は初期化されて便利ですが、登録済みデータを保持したい場合は以下に変更する必要があります。

```properties
spring.sql.init.mode=never
```

## 15. 現在確認できる注意点・改善候補

現状コードから確認できる注意点は以下です。

| No | 内容 | 対応方針 |
|---|---|---|
| 1 | 未ログイン状態で `/medicine` にアクセスすると `account.getId()` が `null` になる可能性がある | `account.getId() == null` の場合は `/login` にリダイレクトする |
| 2 | `/medicine/add`、`/medicine/{id}/edit`、削除処理も未ログイン対策が必要 | 各処理の最初にログイン確認を入れる |
| 3 | 薬更新時に、他ユーザーの薬IDを指定できる可能性がある | 対象薬のユーザーIDとログインユーザーIDが一致するか確認する |
| 4 | 新規登録時にユーザー名の重複チェックがない | `findByName()` で重複確認を追加する |
| 5 | パスワード入力欄が `type="text"` になっている | `type="password"` に変更する |
| 6 | パスワードが平文保存されている | 実務ではハッシュ化が必要 |
| 7 | `medicineAdd.html` と `medicineEdit.html` の `input type=" number"` に空白がある | `type="number"` に修正する |
| 8 | `medicine.html` の `tr` 閉じタグが不足している可能性がある | `th:each` の行を正しいHTML構造に整える |
| 9 | チェックボックスで「済」「未済」を2つ使っている | ラジオボタンに変更すると1つだけ選択できて安全 |
| 10 | 更新ボタンの表示が「追加」になっている | `更新` に修正する |

## 16. 優先して修正した方がよい内容

マスト機能を安定させるため、以下の順番で修正すると安全です。

| 優先度 | 修正内容 |
|---|---|
| 高 | 未ログイン状態で各画面へアクセスできないようにする |
| 高 | 他ユーザーの薬情報を編集・削除できないようにする |
| 高 | 新規登録時にユーザー名の重複チェックを追加する |
| 中 | パスワード入力欄を `type="password"` にする |
| 中 | `type=" number"` を `type="number"` に修正する |
| 中 | チェックボックスをラジオボタンに変更する |
| 中 | `medicine.html` のHTML構造を整理する |
| 低 | 画面タイトルやボタン文言を整える |
| 低 | CSSを調整して見た目を整える |

## 19. 最終まとめ

このアプリは、ログインユーザーごとに薬情報を管理できる服薬管理アプリです。

基本機能として、ユーザー登録、ログイン、ログアウト、薬情報の登録、一覧表示、更新、削除は実装されています。

一方で、未ログイン時の直接アクセス対策、他ユーザーの薬情報を編集・削除できないようにする認可チェック、ユーザー名の重複チェック、パスワード入力欄の改善などは追加対応した方が安全です。

まずは、各Controllerで `account.getId()` が `null` の場合に `/login` へ戻す処理を追加し、ログイン制御を安定させることが重要です。
